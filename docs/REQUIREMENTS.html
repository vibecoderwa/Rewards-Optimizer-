01 · Scope

Scope & goals

A pocket concierge for three credit cards. Glanceable at the door, defensible at the dinner table.

1.1Product goals

  • Tell the user the optimal card for the merchant in front of them, in under 2 seconds, without opening the app.
  • Reconcile each month with what they actually earned vs. what they could have earned.
  • Surface unredeemed credits and perks before they expire — quietly, not noisily.

1.2v1 card set

Amex Gold$3254× dining & groceries
Chase Sapphire Reserve$7953× dining/travel · $300 travel credit · Priority Pass
Capital One Savor$04% dining/entertainment · 3% groceries

1.3Non-goals (v1)

  • Moving money, paying balances, or any action that requires write-access to a financial institution.
  • Recommending new cards the user does not own. (Read-only optimizer, not an affiliate funnel.)
  • Android. iPad. Watch. Web. CarPlay.
02 · Users

Users & personas

Three personas, one app. Each gets a different slice of the product.

Persona 01 · The Optimizer

Owns 2–4 rewards cards, knows their multipliers roughly, forgets them at the register. Comfortable connecting their bank when the value is clear. Gets the full app — pre-swipe guidance, reflective analytics, credit tracking, the whole thing.

Persona 02 · The Manual Adder

Loves the idea of pre-swipe guidance but isn't comfortable linking their full bank account. Willing to tell us which cards they own from a catalog. Gets the location layer — recommendations, push, widgets — but opts out of the analytics layer. The trade-off is shown explicitly at onboarding.

Persona 03 · The Skeptic-Turned-Linker

Starts as a Manual Adder, sees the app working, and upgrades to a bank connection later. Important enough to call out because the upgrade path must exist and must backfill cleanly.

2.1Modes of use

  • Pre-swipe — at the door of a merchant; needs a glance, not a screen. All personas.
  • Reflective — Sunday morning; wants to see the month's tally and which credits are still on the table. Linked personas only.
  • Configurational — one-time setup or a card change; tolerates more friction once. All personas.

03 · Auth

Authentication

Phone number in, OTP back, no passwords. Ever.

FR-AUTH-01

Phone-number sign-in

User enters a phone number with country-code picker. Client validates format before submission. Backend issues a 6-digit OTP via SMS provider.

Must
FR-AUTH-02

OTP verification

Six auto-advancing digit fields, paste-to-fill, 60s resend timer, 3-attempt lockout per code, codes expire in 10 minutes.

Must
FR-AUTH-03

Session tokens

Backend returns short-lived JWT (1h) + rotating refresh token (30d), stored in iOS Keychain. No tokens in AsyncStorage.

Must
FR-AUTH-04

Sign-out & account deletion

Both available in settings. Deletion revokes all Plaid items and purges user data within 30 days; user receives email confirmation if email is on file.

Must
FR-AUTH-05

Biometric re-auth

Face ID / Touch ID gate on app open and before viewing transaction-level detail. Configurable in settings.

Should
FR-AUTH-06

Apple Sign-In

Required by App Store guideline 4.8 if any third-party auth is offered. We're phone-only, so this is a "could" — but watch the review.

Could
04 · Onboarding

Onboarding & bank linking

Bring your wallet, not your card numbers.

FR-ONB-01

Plaid Link as primary path

User taps "Connect bank" → client requests link_token → opens Plaid Link → returns public_token to backend for exchange. Supports multiple institutions per user.

Must
FR-ONB-02

Manual card selection (fallback)

If user declines Plaid, present catalog of supported cards (3 in v1) with multi-select. Saves card identifiers only — no card numbers, no PANs.

Must
FR-ONB-03

Plaid item mapping → owned cards

Backend matches each linked credit account to a known card (by issuer + product detail) and persists the mapping. Unrecognized cards are listed as "tracked, not optimized".

Must
FR-ONB-04

Permissions priming

Pre-permission rationale screens for: Notifications, Location (When In Use → Always), and Plaid data sharing. Each explains the value before the OS dialog fires.

Must
FR-ONB-05

Resumable onboarding

If user drops off mid-flow (e.g. denies Plaid, app backgrounded), resume at last completed step on next launch.

Should
FR-ONB-06

Backfill

On first Plaid link, fetch up to 24 months of historical transactions where available, capped at 5,000 records, paged.

Must
05 · Cards

Card & rewards model

A single source of truth, used identically by client and server.

FR-CARD-01

Shared rewards module

Cards, multipliers, credits and CPP defaults live in core/rewards.ts. Both the mobile client and backend import it. Updates ship via app update; no remote-config required for v1.

Must
FR-CARD-02

Multiplier resolution

For a (card, category) pair, return the earn multiplier as a number plus the unit ("points" or "cash"). Handles category caps (e.g. quarterly, annual).

Must
FR-CARD-03

Cents-per-point (CPP)

Default CPP per card with user-overridable values in settings. CPP feeds the dollar value of points used everywhere in the app.

Must
FR-CARD-04

Credits & perks

Each card has a list of credits with cadence (annual/quarterly/monthly), face value, and a "captured" toggle. The CSR $300 travel credit is treated as a durable, near-100%-captured credit by default.

Must
06 · Transactions

Transaction sync & categorization

Plaid talks. We translate. The user can override.

6.1Sync

FR-TX-01

Webhook-driven sync

Backend handles TRANSACTIONS:SYNC_UPDATES_AVAILABLE webhooks and calls /transactions/sync with the cursor; falls back to a 6-hour cron if a webhook is missed.

Must
FR-TX-02

Idempotent upserts

Adds, modifies, removes from /sync are applied idempotently. Removed transactions are soft-deleted, not hard-deleted, for 30 days.

Must
FR-TX-03

Re-link & ITEM_LOGIN_REQUIRED

When Plaid item enters error state, client surfaces a non-blocking banner and a "fix connection" deep link into Plaid Link update mode.

Must

6.2Categorization

FR-TX-04

15-bucket category map

Map Plaid personal_finance_category.primary + .detailed to dining, groceries, flights, hotels, travel, rideshare, gas, transit, streaming, entertainment, online, shopping, utilities, health, other.

Must
FR-TX-05

Per-transaction override

User can re-categorize any transaction. Overrides persist and improve future suggestions for that merchant (per user, not global).

Should
FR-TX-06

Merchant inference

Group transactions by merchant name + MCC. Show top recurring merchants in the geofence-eligibility list.

Must
07 · Engine

Reward computation engine

Pure functions. No side effects. Same numbers on the phone and on the server.

FR-ENG-01

computeAll(state) → Result

Given user state (transactions, owned cards, credits captured, CPP), returns: net value per card, best card per transaction, what-if delta vs. actual, total annualized.

Must
FR-ENG-02

monthlyTimeline(state)

Returns a 12-month series of (month, realized, optimal) for the timeline chart. Fills empty months with zero, not null.

Must
FR-ENG-03

Best-card-for-merchant

Pure function over (merchantId, basketEstimate?, ownedCards) → { bestCard, multiplier, expectedReward, runnerUp }. Used by geofence push, widgets, and home list.

Must
FR-ENG-04

Determinism

Same input → same output. No Date.now(), no randomness inside the engine. Server unit tests pin behavior with fixtures.

Must
FR-ENG-05

Offline-capable

Client can re-rank cards for a merchant without a network round-trip, given cached card structures and a recent merchant lookup.

Must
FR-ENG-06

Confidence score on every recommendation

Engine emits a confidence value alongside each best-card pick, derived from the gap between #1 and #2 (e.g. 4× vs. 1× = high; 3× vs. 2.5× = low). Surfaced wherever a reco is shown — home list, push, widgets, merchant detail.

Should
FR-ENG-07

Basket-aware uplift estimate

For linked users, when a merchant has ≥3 historical visits, return a basket-aware expected uplift (median basket × multiplier delta × CPP) instead of a generic per-dollar figure. Used in push body copy and merchant detail. Falls back gracefully when no history exists.

Should
08 · Geofencing

Geofencing & location

Quietly aware. Loud only at the door.

FR-GEO-01

Top-merchant geofences

After Plaid sync, derive top 20 recurring merchants by frequency and create geofences (radius 100m) for each via Location.startGeofencingAsync.

Must
FR-GEO-02

Background task

Geofence enter/exit events are handled in a background task and resilient to app suspension. Limit to iOS's 20-region cap; rotate regions based on user location.

Must
FR-GEO-03

"Always" permission rationale

Explicit pre-prompt explaining what we do and don't do with location. Degrade gracefully if user grants only "When In Use" — list works, geofences don't.

Must
FR-GEO-04

Cooldown per merchant

Suppress repeated arrival notifications for the same merchant within 4 hours. Per-merchant override available in settings.

Must
FR-GEO-05

Nearby merchants endpoint

GET /merchants/near?lat&lng&radius returns merchants ranked by best-card uplift, capped at 25 results.

Must
09 · Notifications

Push & live notifications

A whisper, not a klaxon.

FR-NOT-01

Arrival local notification

On geofence enter: schedule a local notification with merchant + best card + expected uplift in dollars. Tap → opens to merchant detail.

Must
FR-NOT-02

Live Activity / Dynamic Island

Start a Live Activity on geofence enter; auto-end after 30 minutes or on geofence exit. Compact view shows card name, expanded shows multiplier & expected reward.

Should
FR-NOT-03

APNs server push for high-value events

Server-issued pushes for: credit expiring within 14 days, card anniversary, large unrecognized transaction. Rate-limited to 2/day per user.

Must
FR-NOT-04

Notification settings

Per-channel toggles: arrivals, expiring credits, weekly recap, anniversary. Quiet hours (default 22:00–08:00).

Must
FR-NOT-05

Dollar-value framing

Notification body always includes a concrete dollar uplift estimate (e.g. "+$9.60 on your usual basket"), not raw multipliers.

Must

Hero summary supports Month / Quarter / Year framing. Persists across sessions per user.

12.5 · Manual

Manual-adder swipe log

A respectful path for the bank-shy. They tell us what they swiped; we give them analytics.

FR-MAN-01

One-tap swipe log

From a merchant detail or arrival screen, manual users can log "I just used [card] here" with a single tap. Optional amount entry, defaults to median basket if known. No bank linking required.

Should
FR-MAN-02

Self-built history

Logged swipes accumulate into a lightweight transaction history scoped to the manual user. Powers a reduced version of realized-vs-potential analytics — best-effort, clearly labeled as user-reported.

Should
FR-MAN-03

Geofence seeding from logs

After a manual user has logged ≥3 swipes at distinct merchants, those merchants seed the geofence list (replacing the "favorites only" fallback in FR-GEO-01).

Should
FR-MAN-04

Upgrade path to linked

A manual user can connect Plaid at any time. On link, the swipe log is preserved and shown as "reported" alongside Plaid-sourced "verified" transactions; duplicates are dedup'd by merchant + date heuristically.

Should

13.5 · Visual principles

Anti-loud aesthetic

Color is information, not decoration. The recommended card pops because nothing else does.

FR-VIS-01

Monochrome by default

Body content, lists, charts and chrome render in ink-on-paper. Accent colors (lemon, mint, coral, sky, plum) are reserved for state — recommendation, on-track, expiring, geo, secondary-accent — never used as decoration.

Must
FR-VIS-02

One accent per screen

At most one accent color is active in a given view. The recommended card row uses the accent; everything else stays neutral. Lists with multiple recos use accent only on the top pick.

Should
FR-VIS-03

No gradients, no decorative color

Solid fills only. No background gradients, no rainbow charts, no color used purely for visual variety. If a UI element needs to feel different, do it with type, weight, or hairline — not hue.

Must
FR-VIS-04

Issuer colors are tertiary

Amex, Chase, and Savor brand colors appear only as small inline marks (a 2px stripe, a label dot) — never as fills, backgrounds, or hero treatments. Keeps the editorial palette intact.

Must
14 · Privacy

Privacy, security & compliance

A finance app on iOS lives or dies on this section.

FR-SEC-01

Server-only Plaid tokens

Plaid access_token never leaves the backend. Encrypted at rest with envelope encryption (KMS-managed key).

Must
FR-SEC-02

TLS & certificate pinning

All client↔server traffic over TLS 1.2+. Cert pinning on the mobile client for the API host.

Must
FR-SEC-03

Plaid Data Transparency Messaging

Use Plaid's verbatim DTM template in onboarding and the App Privacy section.

Must
FR-SEC-04

Location data retention

Raw lat/lng never persisted server-side. Geofence regions are stored as merchant IDs, not user trails.

Must
FR-SEC-05

Account & data deletion

In-app deletion path. SLA: Plaid items revoked immediately; user data purged within 30 days; audit log retained 90 days.

Must
FR-SEC-06

App Privacy disclosures

Apple App Privacy labels accurate to the byte. Financial info, identifiers, location declared with explicit purposes.

Must
FR-SEC-07

No money movement

App copy and metadata never imply funds transfer or account control. Reviewer-facing notes in App Store Connect explicitly state read-only.

Must

16.2Out of scope (v1)
